IL PROBLEMA

La tua history assomiglia a questo?

git log --oneline a8f3c2e Merge branch 'feature/login' into develop
b1d0091 fix: fix del fix
c4e8812 fix typo
d9a1234 WIP
e3c5678 Merge branch 'develop' into feature/login
f7b2345 aggiunto console.log per debug
1a2b3c4 Merge branch 'hotfix/xyz' into develop
2d3e4f5 prova
⚠ Questo è rumore, non storia. Una history illeggibile rende impossibile il code review, il bisect e il rollback chirurgico.

FONDAMENTA

Git è un DAG di snapshot

Ogni commit è un oggetto immutabile identificato dal suo SHA-1. Punta al commit padre e a un tree che rappresenta l'intero snapshot del repo.

Un branch è solo un puntatore mobile a un commit. Spostarlo non cambia i commit — cambia solo dove punta l'etichetta.

struttura di un commit commit a8f3c2e
tree 3f8e1b2
parent d9a1234
author Alice <alice@co.it>

# Il SHA cambia se cambia
# qualsiasi campo — incluso il parent.
# Questo è ciò che fa rebase.

MERGE CLASSICO

Un merge commit preserva la biforcazione

M è il merge commit: ha due parent, C e E. Git registra esattamente quando i rami si sono incontrati.

✓ Vantaggi
Non-destructive. Sicuro su branch condivisi. Preserva il contesto temporale esatto.
✗ Svantaggi
History non-lineare. Log difficile da leggere su progetti con molti branch. Ogni integrazione aggiunge un commit rumore.

GIT REBASE

Riscrive il parent — non il contenuto

Git prende ogni commit di feature, lo riapplica come patch sopra l'ultimo commit di main, e crea un nuovo SHA. D e E scompaiono — nascono D'' e E'' con parent diverso.

⚡ Risultato
History lineare. Ogni commit di feature sembra scritto dopo tutti i commit di main. Fast-forward merge possibile.

CONFRONTO

Merge vs Rebase

Criterio git merge git rebase
History shape Non-lineare — grafo con biforcazioni Lineare — sequenza pulita
SHA dei commit Invariati Riscritti — nuovi SHA
Sicurezza branch shared Sempre sicuro Pericoloso — mai su branch condivisi
Leggibilità log Rumoroso con merge commits Pulito e narrativo
git bisect Più complesso Preciso — ogni commit è atomico
Preserva contesto Sì — timestamp e branch origin Parziale — timestamp originale, parent nuovo

INTERACTIVE REBASE

git rebase -i: il chirurgo della history

apri gli ultimi 4 commit $ git rebase -i HEAD~4

# Nel tuo editor:
pick a1b2c3 feat: aggiungi auth
squash d4e5f6 fix typo
squash 7a8b9c WIP
reword 0d1e2f fix: aggiorna JWT

I comandi disponibili:

pick Usa il commit così com'è
squash Fondilo nel commit precedente
reword Cambia solo il messaggio
edit Metti in pausa per amend
drop Elimina il commit dalla history

SQUASH

Pulisci i commit prima del merge

Prima

D - feat: init login form
E - fix: rimuovi console.log
F - fix: typo username
G - WIP: quasi funziona
H - fix: ora funziona davvero

Dopo squash in D

D'' - feat: aggiungi login form

# Un solo commit atomico,
# messaggio chiaro,
# zero rumore.
Best practice
Fai squash su feature branch prima di aprire la PR. Il reviewer vede una storia pulita, non il tuo processo interno.

GOLDEN RULE

Non fare mai rebase di un branch pubblico

La regola d'oro del rebase:
Non riscrivere mai commit che altri sviluppatori potrebbero aver già scaricato. Se qualcuno ha basato il proprio lavoro sui tuoi SHA e tu li cambi, la sua history diverge.

✓ Sicuro fare rebase

  • Feature branch locale non ancora pushato
  • Branch di cui sei l'unico autore
  • Branch appena pushato e non ancora condiviso

✗ Non fare mai rebase

  • main / develop / release branch
  • Qualsiasi branch su cui altri stanno lavorando
  • Branch da cui altri hanno già fatto pull

FORCE PUSH

--force-with-lease, non --force

Dopo un rebase i SHA cambiano: serve un force push per aggiornare il remote. Ma --force è cieco — sovrascrive qualsiasi commit sul remote, inclusi quelli che altri hanno pushato mentre lavoravi.

⚠ --force distrugge silenziosamente
Se il collega ha pushato su quel branch, il suo lavoro sparisce senza errori.
❌ pericoloso $ git push --force origin feature/x

✓ sicuro $ git push --force-with-lease origin feature/x
--force-with-lease
Controlla che il tuo ref locale del remote coincida con il remote reale. Se qualcuno ha pushato nel frattempo, fallisce — proteggendoti.

TEAM WORKFLOW

Il ciclo rebase nel lavoro quotidiano

1
git checkout -b feature/xyz — crea branch locale da main aggiornato
2
Sviluppa liberamente, commit anche sporchi — sono tuoi
3
git fetch origin && git rebase origin/main — allineati a main
4
git rebase -i origin/main — squash + reword: prepara PR pulita
5
git push --force-with-lease — aggiorna il remote in sicurezza
6
PR approvata → fast-forward merge su main — nessun merge commit

CONFLITTI

Il rebase si ferma — non paniccare

il rebase si ferma su conflitto CONFLICT (content): Merge conflict in auth.ts
error: could not apply d4e5f6...
# Risolvi → poi:
$ git add auth.ts
$ git rebase --continue
oppure, se vuoi annullare tutto $ git rebase --abort
# Torni esattamente dove eri prima

Con il rebase i conflitti appaiono commit per commit: ogni stop è un conflitto su un commit specifico — più chirurgico di un merge che accorpa tutto.

Tip: usa git rerere
git config --global rerere.enabled true
Git memorizza come hai risolto ogni conflitto e lo riapplica automaticamente nei rebase successivi sullo stesso codice.

PITFALL

Errori classici e come evitarli

🔴 Rebase di main

Riscrivere main rompe la history di tutti. Proteggilo con branch protection rules su GitHub.

🔴 Squash di merge commit

Non fare squash di commit che includono merge. Usa rebase --rebase-merges se la struttura serve.

🟡 Perdere commit dopo abort

I commit "persi" sono nel reflog per 90 giorni: git reflog + git checkout SHA.

🟡 Tags che puntano ai vecchi SHA

Dopo un rebase, i tag annotati continuano a puntare ai vecchi commit. Non fare rebase di commit già taggati.

🟡 Rebase con submodules

I submodule non vengono aggiornati automaticamente. Usa --recurse-submodules.

✓ Recovery universale

git reflog registra ogni movimento di HEAD. Puoi tornare a qualsiasi stato degli ultimi 90 giorni.

DECISIONE

Quando usare cosa — regole pratiche

Usa Rebase quando…

Stai allineando il tuo feature branch a main prima della PR
Vuoi pulire i commit WIP prima che li vedano altri
Vuoi fare fast-forward merge su main senza merge commit
Hai bisogno di git bisect efficace su bug regressions

Usa Merge quando…

Vuoi preservare l'evidenza di una collaborazione parallela
Stai integrando un release branch o hotfix su main
Il branch è condiviso con altri sviluppatori
Vuoi un audit trail chiaro di quando le feature sono entrate

CHEAT SHEET

Comandi essenziali

rebase & interactive # allinea branch a main
$ git rebase main

# rebase interattivo ultimi N commit
$ git rebase -i HEAD~N

# continua dopo conflitto
$ git rebase --continue

# annulla tutto
$ git rebase --abort

# force push sicuro
$ git push --force-with-lease

# abilita rerere
$ git config --global rerere.enabled true

# recupera commit "persi"
$ git reflog
$ git checkout <SHA>
